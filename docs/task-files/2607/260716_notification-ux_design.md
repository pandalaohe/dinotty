## 1. Positioning & problem
Notification = low-noise "there is something to continue here, come over" signal, NOT an alarm. Today (verified): `useNotification.ts:138-199 handleEvent` fires on EVERY bell/notify event with no focus guard (beeps while you watch); unread lives only in per-surface frontend memory (`unreadByPane`), never synced; read clears only on tab-switch/close; all presentation settings are server-shared; the notification WS is downstream-only; two pre-existing bugs sit in the path (dead `set_debounce_ms`, Lagged silent-death). NUX1 makes attention state backend-authoritative + cross-end synced, moves presentation local + per-surface, and adds active-read + low-noise controls.

## 3. Architecture — two-layer split
- **Attention state (backend-authoritative, in-memory, cross-end synced):** whether each pane (and each pane-less plugin notif) is unread, aggregate severity, firstUnreadAt. Owned by `NotificationBroadcast`'s new `AttentionLedger`. Synced over the (now bidirectional) `/ws/notify` as per-revision `state_delta` transactions.
- **Presentation (per-surface, local):** popup / sound / vibrate / tab-indicator channel, DND level (E1), ignore-current-tab (D2), quiet-hours (E5), coalesce (E2). Each surface reads its own localStorage settings and decides how to PRESENT an authoritative event. Never synced.
- **Authoritative UI vs presentation (audit B9):** the attention DOT / badge / count reflect the authoritative ledger and update on every `state_delta` — NOT subject to the presentation gate — EXCEPT the active-read suppression below. The presentation gate governs only popup/sound/vibrate + the local `tab_indicator` toggle. "silent" (E1) still shows the authoritative dot.
- **Active-read is authoritative-state-driven, not `raised`-driven (codex-r3 C3-02):** when an incoming `state_delta`/`snapshot` marks a pane unread AND that pane is the current active+foreground leaf (§8), the surface (a) SUPPRESSES the dot for that pane at apply-time (no flash) and (b) emits an `active_observed` `mark_read`. This is evaluated on EVERY authoritative-state application, so a lost `raised` presentation cue never leaves an active pane stuck unread. `raised` drives ONLY popup/sound/vibrate presentation, never the read transition.

Central flow:
```
terminal bell / plugin notify / osc notify
  -> backend applies INGEST gates by SOURCE TYPE (§10 table) BEFORE assigning seq; hooks run only on accepted events
  -> backend records unread event (assigns global-per-epoch eventSeq, exact occurred_at), bumps revision R
  -> /ws/notify broadcasts state_delta@R (ALL pane/notif changes for R, one envelope) + a `raised` presentation cue
  -> each surface applies state_delta@R (revision-gated) -> authoritative attention UI; if target is active+foreground -> suppress dot + emit active_observed (§8)
  -> each surface runs presentationGate() locally on `raised` -> popup/sound/vibrate or suppress

focus / first keystroke / tab activate / tab close / goto / active-observed / dismiss / clear-all
  -> surface sets an OPTIMISTIC overlay keyed by requestId (per target, carries the declared watermark/notifId) + cancels pending coalesce
  -> sends notification.mark_read{v, epoch, clientId, requestId, panes:[{paneId, throughEventSeq}], notifs:[{notifId}], reason}
  -> backend RESERVES (clientId,requestId) under the ledger lock (dedup), applies per target, advances watermark, bumps revision Rn
  -> backend replies mark_read_result{requestId, epoch, appliedAtRevision:Rn|null, results:[{target, status}]} (never dropped, §5 control lane)
  -> backend broadcasts state_delta@Rn to ALL surfaces
  -> sender retires each overlay target once BOTH result=applied AND authoritative cache has reached appliedAtRevision
     (no-op already-read target: applied@currentRev -> retires once cache ≥ that revision, typically already true; stale_epoch/invalid/not_found: drop that target + re-adopt authoritative state)
```

## 4. Backend AttentionLedger (`src/notification.rs`)
`NotificationBroadcast` gains an in-memory ledger; all mutations go through its methods so ledger-mutation + outgoing `state_delta` are one atomic op (callers stop sending on the raw broadcast tx). Every mutation bumps `revision` and emits exactly one `state_delta` carrying all changes at that revision. ALL mutation paths — client `mark_read`s, producer POSTs, the TTL sweep, identity-cap eviction, pane-close GC — AND `snapshot` capture serialize through this ONE ledger lock; a snapshot is captured and enqueued onto the client's outbound stream while the lock is held (codex-r4 C4-01 / claude-r4 N5).

```
AttentionLedger
  epoch: String            // boot-scoped ledgerId, fresh per process start
  next_event_seq: u64      // GLOBAL within one epoch; monotonic checked increment (no wrap)
  revision: u64            // monotonic; every state mutation bumps it
  panes: HashMap<PaneId, PaneAttention>
  notifs: HashMap<NotifId, NotifAttention>   // pane-less plugin notifs (R11)
  dedup: HashMap<(ClientId,RequestId), DedupEntry>   // §5 atomic reservation
PaneAttention {
  read_through_seq: u64,
  latest_event_seq: u64,               // highest seq recorded for THIS pane (exposed in state_delta)
  unread: VecDeque<UnreadEvent>,       // EXACT per-event, oldest→newest; see cap/TTL below
  read_at: Option<ms>,                 // set when fully read; drives GC retention
}
NotifAttention {                       // pane-less plugin notif (R11)
  event_seq: u64, occurred_at: ms, severity: Severity,
  read: bool, read_at: Option<ms>,
}
UnreadEvent { seq, occurred_at, severity }   // EXACT — never merged in a way that loses occurred_at
DedupEntry { reserved_at: ms, done: bool, outcome: Option<DedupOutcome>, payload_hash: u64 }
// DedupOutcome = MarkRead(MarkReadResult)
//              | Producer(accepted{notifId?|paneId, eventSeq, revision} | suppressed{reason} | not_found)
//   — one cached shape per consumer, so a producer replay gets its EXACT prior response (codex-r4 C4-04)
```
- **eventSeq / read watermark (R3, union-of-reads):** `eventSeq` is global within one epoch; every accepted event increments it, updates the target pane's `latest_event_seq`, and pushes an exact `UnreadEvent`. `mark_read(pane, N)` sets `read_through_seq = max(existing, N)` after validation and pops unread events `seq ≤ watermark` from the head → clients only advance; stale/offline receipts idempotent; a new event after full-read re-marks unread (new signal).
- **Reject-not-clamp (audit B3):** a `mark_read` target is REJECTED (`invalid`, no mutation) when `throughEventSeq > pane.latest_event_seq`. Clients only send a seq they observed in a `state_delta` for the current epoch. `state_delta` carries `latestEventSeq` per pane so a client always knows the ceiling.
- **firstUnreadAt (R9, E8) — EXACT:** = `occurred_at` of the deque head (first remaining unread after popping ≤ watermark). Always a real retained event's timestamp; exact when an older read leaves newer unread; survives reload/reconnect; identical cross-surface; feeds SVT1.
- **Bounded memory — cap + TTL, NOT lossy merge:**
  - Per-pane unread deque capped at `UNREAD_CAP`. Overflow drops the OLDEST event (head) + logs `[unread-drop]`; firstUnreadAt becomes the next head (still EXACT, slightly newer).
  - Retention TTL: a lazy sweep (mutation touch + periodic tick) expires unread events older than `UNREAD_TTL`. When a sweep (or read) empties a pane's deque it TRANSITIONS TO READ: `read_through_seq = latest_event_seq` + `read_at` set (codex-r3 C3-06 — makes the fully-read predicate true, so GC is eligible). The SAME `UNREAD_TTL` sweep applies to a `NotifAttention` (its single `occurred_at`): an unread pane-less notif older than TTL flips `read=true` (claude-r3 F5 — no separate stale-unread accrual, one-shot).
  - Identity cap: `panes`/`notifs` maps capped at `IDENTITY_CAP` each. Eviction order when OVER cap (codex-r3 C3-06, mixed-state): (1) evict fully-read entries oldest-`read_at`-first — even if their retention TTL has NOT elapsed (the cap is a HARD bound; retention TTL is best-effort); (2) only if still over cap with no fully-read entries left, evict the live-unread entry with the oldest firstUnreadAt (ignored longest) + log `[identity-evict]`. Severity aggregated per identity (max of retained unread) — the only lossy aggregation; never touches firstUnreadAt.
  - Card BODIES (title/body text) bounded SEPARATELY from attention metadata (the 100-cap history is a presentation concern).
- **Pane / split-leaf close (codex-r3 #3):** closing a pane removes its `PaneAttention` (revision bump + `state_delta` `paneId` removal); unread does not migrate to siblings. The R2 tab-close read-trigger (§7.2, graceful close marks read) and this close-GC (identity gone) are distinct.
- **Plugin notif target model (codex-r3 C3-04):** a plugin `ui.notify` WITHOUT `paneId` enters `notifs` as a pane-less identity (read via `notifId`); a plugin notify WITH a valid `paneId` records as a normal PANE event on that pane (read via that pane's watermark) — it does NOT create a `notifs` entry. Exactly one target model per request; the emitted `state_delta`, count, dismiss/clear-all/goto target follow from which one.
- **Boot epoch (audit R-c):** `epoch` fresh each start; every `state_delta`/`snapshot` + every `mark_read` carries it; server rejects (`stale_epoch`) a foreign-epoch read; client resets baseline on a new epoch. In-memory only: a backend restart LOSES unread + firstUnreadAt (accepted, §13); R9 covers reload/reconnect only.
- **Wire numerics:** `eventSeq`/`revision`/seqs are decimal STRINGS on the wire; clients compare via BigInt (numeric), never JS string/lexicographic (audit codex-r2 #1).

## 5. WebSocket protocol (`/ws/notify`, `src/ws/mod.rs`)
Extend `/ws/notify` to bidirectional; `/ws/sync` untouched. Rationale (cross-suggest B-stronger): notification events + their attention-state revisions share ONE ordered channel → no cross-socket ordering race.

- **Connect-time protocol version gate (codex-r3 C3-09, D-deploy-A):** every client sends its protocol `v` at connect (WS query param or first hello frame; transport confirmed at implement, §14). FAIL-CLOSED classification (codex-r4 C4-08 / claude-r4 N3): a `v` that is ABSENT, malformed, or unparsable is treated as version 0 — a genuinely pre-NUX1 client never sends `v` at all, so "absent" IS the legacy case — and rejected identically to below-minimum. Rejection is delivered as a WS CLOSE frame with a dedicated close code + reason (`protocol_upgrade_required` is a CLOSE FRAME, not a JSON envelope — a pre-upgrade client needs no new parsing to receive a close); the upgraded web surface auto-reloads on that close code; the upgraded Tauri desktop shows an update prompt; a pre-upgrade client simply sees its socket closed and its reconnect re-refused (never silently degraded). This PREVENTS a persistent old desktop binary from silently connecting and receiving only legacy `raised` while never participating in authoritative state. No capability handshake is built; the gate is a single min-version check.
- **Transport FIX — bounded per-client queue + resync (audit B2):** the current forwarder `while let Ok(event)=rx.recv().await` (`ws/mod.rs:868`) over bounded `broadcast(256)` silently EXITS on `RecvError::Lagged`. Fix: each client gets a BOUNDED outbound queue (message-count AND byte cap). On overflow, drop queued `state_delta`s and enqueue a single `resync_required`; when capacity returns, send a fresh authoritative `snapshot`. A client that cannot drain within a bounded interval is disconnected (close → reconnect). Never `mpsc::unbounded`. `Closed` terminates; nothing else silently exits. (The pure "stop silent-exit, disconnect on Lagged" half ships as standalone P1, §12; `resync_required`/`snapshot` ship in the atomic P3+P4 bundle.)
- **Control lane (claude-r3 F2; membership + overflow codex-r4 C4-05) [AMENDED at P3b implement, see §15]:** control lane = `mark_read_result`, `resync_required`, `stale_epoch` — exactly these (`protocol_upgrade_required` is a close frame, above). They travel a SEPARATE, small, bounded control queue that data overflow never touches — `state_delta` overflow drops only deltas. If the CONTROL queue itself fills (a client not draining while still sending reads), the server DISCONNECTS that client immediately (close → reconnect) — never blocks the ledger, never silently drops an ack, never grows unbounded; safe because dedup replays cached outcomes after reconnect and the reconnect snapshot re-bases state. Complement (belt-and-braces): the client also runs an ack-timeout — an unacked `mark_read` is re-sent (idempotent by `(clientId,requestId)`) after a bounded interval EVEN WITHOUT a reconnect, with a BOUNDED retry count (exhaustion → drop the overlay + re-adopt authoritative state). Snapshot re-validation of pending overlays is FORWARD-ONLY (codex-r4 C4-07): a fresh `snapshot` may RETIRE an overlay when it PROVES the read (`readThroughSeq ≥` overlay watermark / notif read) but NEVER rolls one back early — a snapshot captured before a pending request commits can still show the target unread without disproving it; rollback happens ONLY on an explicit reject result or ack-timeout exhaustion.
- **Resync / revision reconciliation (codex-r3 C3-01):** two rules close the equal-revision hole: (1) a `snapshot` is a FULL authoritative replace applied whenever `revision ≥ cacheRevision` (`≥`, not `>`) — it can never be rejected for equal revision; (2) after a client receives `resync_required` it enters `awaiting_snapshot` and DISCARDS interim `state_delta`s until the fresh `snapshot` is adopted (the snapshot is complete, so buffered deltas are redundant); (3) ORDERING INVARIANT (codex-r4 C4-01): snapshot capture + enqueue happen under the ledger lock (§4) and each client receives ALL state messages over ONE FIFO stream — so any delta enqueued after `snapshot@N` has `revision > N` and ARRIVES after it; `awaiting_snapshot` can only ever discard deltas the incoming snapshot already covers, and a genuinely newer delta arrives after snapshot adoption (client back in normal state) and applies normally — no permanent staleness. `state_delta`s still apply on strict `revision > cacheRevision` in the normal (non-awaiting) state.
- **state_delta transaction envelope (audit codex-r2 #1):** each revision emits ONE `state_delta { epoch, revision, panes:[{paneId, latestEventSeq, readThroughSeq, firstUnreadAt, severity, removed?}], notifs:[{notifId, read, ...}|{notifId, removed}] }` carrying ALL targets changed at that revision. A multi-pane `mark_read` at one revision arrives as ONE delta with the SAME revision for every target — clients accept all targets in a `revision > cacheRevision` delta together (never per-target strict-`>`). `cacheRevision` advances ONLY from `snapshot`/`state_delta`, NEVER from `mark_read_result`.
- **Envelope dispatcher (audit R-b), server→client** (dispatched BEFORE legacy `handleEvent`): `raised` / `state_delta` / `snapshot` / `epoch` / `mark_read_result` / `resync_required` (`protocol_upgrade_required` is a close frame, not an envelope — above). `raised` keeps the old `type:"bell"|"notify"` field the pre-upgrade `handleEvent` understands AND adds a `v` tag + new fields; the new dispatcher intercepts first. (Legacy safety for state envelopes is provided by the connect-time version gate above — an un-upgraded client is closed before it can mis-handle a `state_delta`, so the old code never needs to understand new envelopes.)
- **Client→server:** `notification.mark_read { v, epoch, clientId, requestId, reason, panes:[{paneId, throughEventSeq}], notifs:[{notifId}] }`. `clientId` = stable per-surface-install id (localStorage), stable across reconnects. `requestId` = GLOBALLY unique per client across BOTH ws-reads and producer POSTs, minted with a tab-nonce so two same-origin tabs sharing a `clientId` never collide (codex-r3 C3-03). `reason ∈ {focus, terminal_input, tab_activate, tab_close, pane_close, goto, active_observed, dismiss, clear_all}`. Parsed in the existing Text branch (single reader).
- **Atomic dedup (codex-r3 C3-03) [in-flight semantics AMENDED at P3b implement, see §15]:** the `(clientId,requestId)` entry is RESERVED under the ledger lock BEFORE any mutation. A duplicate arriving while the original is in-flight BLOCKS on the reservation and, when the original completes, returns the EXACT cached per-target result + original `appliedAtRevision` (never a second mutation). A completed entry replays its cached outcome. Same key with a DIFFERENT `payload_hash` → `conflict` result (no mutation). `POST /api/notify` shares this table, so a replayed producer request never allocates a second `notifId`/`eventSeq` within the retry horizon below. **Reservation lifecycle (codex-r4 C4-03 / claude-r4 N1):** reserve → ledger mutation → outcome install (`done=true`) execute as ONE critical section DETACHED from the requesting connection's task lifetime — a client disconnect or WS-task cancellation can lose only the REPLY, never abort the mutation half-committed. Gate-suppression (§10) is evaluated INSIDE this reserved section and the `suppressed` outcome IS cached — a same-key retry replays `suppressed` even if settings changed since (a re-attempt needs a fresh requestId). `DEDUP_TTL`'s clock starts at COMPLETION (`done=true`), never at `reserved_at` — an in-flight entry is never TTL-evicted; a SEPARATE bounded `RESERVATION_TIMEOUT` reclaims an entry still `done=false` past it (bug-class stall backstop): blocked waiters get a retryable error and the key may be freshly re-reserved. `DEDUP_TTL` is also the producer RETRY HORIZON (documented in the §5 endpoint contract): a caller must not replay a requestId older than `DEDUP_TTL` — beyond it the key may be evicted and a replay records anew (codex-r4 C4-04).
- **Ack/reject state machine — PER-TARGET (audit B1 + codex-r2/r3):** every `mark_read` gets `mark_read_result { requestId, epoch, appliedAtRevision, results:[{ target:{paneId|notifId}, status: applied | stale_epoch | invalid | not_found | conflict }] }`. `appliedAtRevision` = the revision the accepted targets landed at; for a NO-OP request whose targets were ALL already read (valid but no mutation) status is `applied` and `appliedAtRevision = current revision` (codex-r3 C3-05) — the overlay retires once the sender's cache reaches that revision, TYPICALLY already true; if the ack overtakes the state stream the overlay simply waits for the ≥-revision delta/snapshot (not literally immediate — codex-r4 C4-09); for an ALL-REJECTED request (no accepted target) `appliedAtRevision = null` and every overlay target is dropped per its reject status. Per accepted target: overlay retires only once the cache reaches `appliedAtRevision`; on `stale_epoch` cancel ALL old-epoch pending + adopt snapshot; on `invalid`/`not_found`/`conflict` drop that target + re-adopt authoritative state. Dedup by `(clientId, requestId)`; a client resends only unacked requests (ack-timeout or after reconnect) with the CURRENT epoch.
- **Plugin producer endpoint (R11) — COMMITTED (audit codex-r2 #4):** evolve existing `POST /api/notify` (`src/main.rs:743`) — no parallel route. Contract: `POST /api/notify { clientId, requestId, source:"plugin", paneId?, type, title?, body }` (same auth as today). Responses: accepted → `200 { notifId?|paneId, eventSeq, revision }` (`notifId` only for the pane-less path, §4 target model); gate-suppressed by `notification.enabled` → `200 { status:"suppressed", reason }` (no eventSeq); invalid `type`/malformed id → `400`; well-formed but UNKNOWN/closed `paneId` → `404 { status:"not_found" }` — the event is NOT recorded: no ghost `PaneAttention`, never silently downgraded to pane-less (codex-r4 C4-10 / claude-r4 N4); dedup payload conflict → `409`. EVERY outcome (incl suppressed/not_found) is dedup-cached (§4 DedupOutcome). Server applies §10 `plugin`-source gates, dedups atomically, records ONCE, bumps revision, broadcasts `state_delta`. The initiating surface does NOT locally insert — it waits for its own broadcast. Legacy callers without `clientId/requestId` still work (server mints a one-shot id, no cross-end dedup).

## 6. Frontend reconcile (`useNotification.ts`) [history semantics AMENDED post-review, see §17]
- **Envelope split:** an onmessage dispatcher routes `snapshot`/`state_delta`/`epoch`/`mark_read_result`/`resync_required` to reconcile logic and `raised` to presentation (the version-gate rejection arrives as a WS close code, handled by the close/reconnect handler); the old `handleEvent` no longer sees state messages.
- **Authoritative cache + optimistic OVERLAY (audit B1 + codex-r2/r3):** the authoritative cache is updated only by `snapshot`/`state_delta`: a `state_delta` applies on `revision > cacheRevision` (BigInt compare) in the normal state; a `snapshot` applies on `revision ≥ cacheRevision` and wholesale-replaces; after `resync_required` the client is `awaiting_snapshot` and discards interim deltas (§5). On a new `epoch`, discard baseline + adopt snapshot. A local read writes a SEPARATE optimistic overlay KEYED BY requestId, one entry per target, **carrying the declared `throughEventSeq` (pane) / `notifId`** (claude-r3 F1). Masking rule: an authoritative unread entry is hidden ONLY IF an overlay target covers it — for a pane, only unread AT-OR-BELOW the overlay's `throughEventSeq`; for a notif, only that exact `notifId`. Any authoritative unread ABOVE the overlay watermark (a genuinely new event) STAYS VISIBLE. **Masked-projection scope (codex-r4 C4-06):** the mask decides unread PRESENCE exactly (`latestEventSeq ≤ watermark` → pane shows read; `>` → unread); the pane AGGREGATE fields (`firstUnreadAt`, severity) are NOT re-derivable client-side while a mask hides part of the unread set (the wire carries no per-event list) — during the pending window they may transiently reflect the pre-read authoritative values. ACCEPTED TRANSIENT (same class as §8's), corrected by the server's post-apply `state_delta` within one ack+delta round-trip; SVT1 ordering uses the authoritative values throughout. `mark_read_result` retires/rolls back overlay targets per §5; snapshot re-validation is FORWARD-ONLY (§5 — retire on proof, never early-rollback).
- **Active-read on state application (codex-r3 C3-02):** when applying a `state_delta`/`snapshot` that marks the active+foreground leaf unread, suppress its dot + emit `active_observed` (§3/§8) — independent of `raised`.
- **firstUnreadAtByPane** comes from backend `state_delta` (exact), not derived locally.
- **Idempotent insert (audit R-g):** notification-history dedup key = `epoch + paneId + eventSeq` for pane events, `epoch + notifId` for pane-less plugin notifs (claude-r4 N2); epoch required — seqs reset per epoch. Today `:164-173` mints a fresh local id per delivery → double-insert on snapshot replay.
- **Count semantics (audit B-NB3):** replace `unreadCount = notifications.length` (`:92-95`) with `unreadAttentionCount` = unread panes + unread pane-less notif identities; keep `historyCount` separate. Badge (`App.vue:76`), panel visibility, tests, aria labels audited.

## 7. Read triggers → all emit backend mark_read (R1/R2/R3/R10/R11/R12)
All paths set the optimistic overlay (with declared watermark) AND send `mark_read`. Foreground guard (R1): `focus` and `terminal_input` apply the same `isAppForeground` predicate as active-read (§8).
1. **Manual tab click / activate (B):** `activateTab → clearForPaneIds` (App.vue:792) — `mark_read` for all leaves, each with its OWN `throughEventSeq` (audit R-l), reason `tab_activate`.
2. **Tab close (B):** reason `tab_close` — sent only AFTER backend close succeeds (audit NB1): keep prior attention pending, roll back if `closeTab` (App.vue:882/896) fails.
3. **Notification goto (F):** `revealPane` (§10) marks the revealed tab read, reason `goto`.

## 8. Active-read model (R12 / E-active / E4)
"Active" = pane is the current focused pane (`tab.activePaneId`) AND the app window is foreground. Rules:
- A `state_delta`/`snapshot` marking an ACTIVE pane unread → auto-read: SUPPRESS its dot at apply-time AND emit `mark_read` reason `active_observed` (§3/§6/§7.6) so all surfaces converge — evaluated on authoritative-state arrival, so a lost `raised` cannot leave it unread (codex-r3 C3-02).
- Any inactive pane, OR any pane while the window is backgrounded/hidden → notify per presentation gate on `raised`. (E4 fix.)
- **`isAppForeground` per-surface adapter (audit B4):** web = `document.visibilityState`/`document.hasFocus()`; Tauri = native window focus/blur events (NOT `document.hasFocus()`) — shared test vectors, real-window parity at implement (§14). Re-evaluate on `visibilitychange`/`focus`/`blur`.
- **Ordering:** active state is sampled when the authoritative `state_delta` is APPLIED. A remote inactive surface may briefly present on `raised` before the active surface's `active_observed` read broadcasts back; accepted transient (low-noise) unless implement finds it jarring → short causal-suppression window fallback.
- Composes with the presentation gate: the §9 active-leaf mask suppresses ALL emit channels for the active leaf's own `raised` (codex-r4 C4-02 — the state-side auto-read here handles the dot + cross-end read; the mask handles popup/sound/vibrate); D2 governs the active tab's OTHER (unfocused) leaves.

## 9. Presentation gate (per-surface, local) [popup/toast semantics AMENDED post-review, see §17]
A single `presentationGate(event, ctx)` produces a deterministic OUTPUT VECTOR (audit B9 + codex-r2/r3 #6):
```
{ storeHistory, showTabIndicator, showPopup, playSound, vibrate }   // dot/badge NOT here — authoritative (§3)
```
Pipeline = **base(settings) → E1 mask → active-leaf mask → D2 mask → E5 mask → E2 coalesce-schedule**:
- **Base vector from settings:** `presentation_enabled=false` → all channels off (authoritative dot remains). Else each channel starts from `channels.{sound→playSound, vibration→vibrate, panel→showPopup, tab_indicator→showTabIndicator}`; `storeHistory` from panel/history setting. E1/D2/E5 can only turn base channels further OFF.
- **E1 (R6) mask:** normal = keep base; dot+sound = clear `showPopup + vibrate` (keep sound — codex-r3 C3-08, the level is literally "dot+sound"); silent = clear `showPopup + playSound + vibrate` (indicator+history+dot remain).
- **Active-leaf (R12) mask (codex-r4 C4-02):** if the event's pane IS the current focused leaf AND `isAppForeground` (§8 predicate, sampled at gate time) → clear ALL emit channels (`showPopup + playSound + vibrate`); `storeHistory`/indicator unaffected. This enforces R12's "no dot/sound" inside the presentation pipeline itself — the §3/§8 active-read handles the state side; coalesce-cancel is NOT relied on for this.
- **D2 (R5) mask:** if the event's pane is in the active tab (whole tab incl leaves) AND ignore-current-tab is ON (default) → clear `showPopup` only — in practice D2 only ever fires for the active tab's OTHER (unfocused) leaves; the active leaf is already fully masked above. Sound/vibrate stay governed by base + E1 (no D2 sound field — audit F5).
- **E5 (R8) mask:** quiet-hours clears `showPopup + playSound + vibrate` (NOT indicator/history/dot). Half-open `[start, end)`; cross-midnight (`start>end`): quiet iff `now≥start OR now<end`; normal (`start<end`): quiet iff `start≤now<end`; **`start==end` = never quiet** (audit F4/B9). Evaluated from CURRENT local civil time at present time; re-evaluate on visibilitychange/focus (DST/tz safe).
- **E2 (R7) — coalesce SCHEDULER over the WHOLE surviving vector (codex-r3 C3-07):** after masking, E2 defers + collapses a burst of same-pane presentations within `presentation.coalesce_window_ms` (local field) into ONE presentation across ALL surviving emit channels (popup, sound, AND vibrate) — so 10 rapid bells fire at most one popup + one sound + one vibrate (satisfies R7's "one reminder" for every channel). `storeHistory` is NOT coalesced — it runs per ledger event (authoritative history is complete). The collapsed presentation uses the HIGHEST-severity event in the window as representative (tie → latest). Timer keyed per pane; a mark-read for that pane cancels its pending coalesce; a read before flush yields nothing; a later event above the read watermark stays eligible; mid-window settings change re-evaluated at fire. Does NOT drop ledger events (dot/count/firstUnreadAt already authoritative).
- **Debounce vs coalesce — DECOUPLED (audit B8):** `set_debounce_ms` → `bell.ingest_debounce_ms`, a SEPARATE backend ingest-debounce that drops duplicate bells BEFORE ledger ingest, globally.
